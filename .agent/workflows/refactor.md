# リファクタリング・ワークフロー

**トリガー:** `/refactor` (またはリファクタリングが必要と判断された場合)

**説明:** 振る舞いを変えずにコードの品質を安全に向上させるための手順です。

## 1. 準備と計画
- **ターゲットの特定:** 以下の「コードの臭い」がある箇所を特定します。
    - 長すぎるメソッド（20行超）
    - 巨大なクラス（SRP違反）
    - 重複したコード
    - 深すぎるネスト（ガード節で解決可能）
- **依存関係の確認:** `GEMINI.md` の「プロジェクト構造認識」に基づき、影響範囲（参照元）を確認します。
- **テストの有無:** テストがない場合は、まずテストを作成してください。

## 2. 実装ステップ (Incremental Steps)
- **最小単位での実行:** 一度に一つの変更（関数抽出、名前変更など）だけを行います。
- **検証の徹底:** 各ステップの後に必ず `pnpm verify` を実行し、既存のテストがパスすることを確認します。
- **コミットの推奨:** 成功するたびに進捗を確定させます。

## 3. リファクタリング・チェックリスト
- [ ] 振る舞い（挙動）を変えていないか？
- [ ] テストはすべてパスしているか？
- [ ] ネストは浅くなったか？
- [ ] 命名は意図を明確に表しているか？
- [ ] 不要な重複は削除されたか？

## 4. 代表的なパターン
- **関数/変数の抽出:** 複雑なロジックに名前を付けて切り出す。
- **ガード節の導入:** `if-else` のネストを早期リターンで平坦化する。
- **パラメータ・オブジェクト:** 多すぎる引数をオブジェクトにまとめる。

**システム命令:**
このワークフローが呼び出された際、あなたは「品質の管理人」として振る舞ってください。速度よりも「壊さないこと」と「読みやすくなること」を最優先してください。
