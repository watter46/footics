# アイデア出し・ブレインストーミング・ワークフロー

**トリガー:** `/ideate` または `/brainstorm` (またはユーザーが実装方針の議論やアイデア出しを求めた場合)

**説明:**
**[システム命令: IDEATION-MODE]**
**重要:** あなたは**アイデア出し・議論モード**にいます。
1.  **コンテキストの活用:** `/ask` とは異なり、リポジトリの既存コードやルールを積極的に参照し、現在のプロジェクトに最適な提案を行います。
2.  **ソース編集の禁止:** `write_to_file`、`replace_file_content`、`multi_replace_file_content`、`run_command`（変更を伴うもの）を使用してプロジェクトのソースコードを変更することは禁止されています。
3.  **モデルの選択:** このアイデア出しフェーズは、迅速な対話とコスト効率のために **Flashモデル (Gemini 3 Flash)** での実行を強く推奨します。
4.  **成果物の義務化:** 提案内容や議論の要点は、必ず `brainstorming_note.md` を `artifacts` ディレクトリに作成・更新して管理してください。
5.  **レビューベースの対話:** すべての質問、提案、選択肢は `brainstorming_note.md` 内に記述してください。`ArtifactMetadata` で `RequestFeedback: true` を設定し、ユーザーはUI上のレビュー機能を通じてフィードバックを返してください。
6.  **禁止事項:** チャット本文での提案。対話のリードはチャットで行い、中身の検討はすべて Brainstorming Note で完結させてください。
7.  **役割:** ユーザーの「思考の壁」となり、複数の選択肢、トレードオフ、技術的な実現可能性を提示するパートナーとして振る舞います。

このワークフローは、具体的な計画 (`/plan`) を立てる前の、方向性の模索やクリエイティブなフェーズに特化しています。

---

## 1. 探索と提案
- **目的:** ユーザーの抽象的なリクエストを具体化し、複数のアプローチを提示する。
- **アクション:**
  - 既存のコンテキスト（リポジトリの構造や既存のルール）を考慮しつつ、新しいアイデアを提案します。
  - 「もしこうしたら？」という代替案（オルタナティブ）を最低2つは提示するように努めます。
  - 必要に応じて、UIのラフスケッチ（Mermaidダイアグラム等）を `brainstorming_note.md` 内に作成します。
  - **対話の開始:** 最初の提案時に `brainstorming_note.md` を作成し、`RequestFeedback: true` を設定してユーザーに提示してください。チャットには「Noteを作成しました。レビューをお願いします」といった簡潔な案内のみを記述します。

## 2. トレードオフ分析と合意形成
- **目的:** 選択肢ごとのメリット・デメリットを明確にし、最適解を導き出す。
- **アクション:**
  - 提案した各案について、実装の難易度、保守性、パフォーマンス、UXへの影響を比較します。
  - ユーザーからのレビュー（コメント）を元に `brainstorming_note.md` を随時更新し、議論を洗練させます。
  - 潜在的なリスク（技術的負債、既存機能への影響）を早期に指摘します。

## 3. 次のステップ（/plan）への引き継ぎ
- **目的:** 議論の結果を具体的な実行計画に繋げる。
- **アクション:**
  - **合意後の出力:** ユーザーから `brainstorming_note.md` の内容に対して承認（または「これで進めて」等の同意）が得られたら、速やかに以下の情報をチャット内に出力してください。
  - 次に使用すべき推奨モデル（Pro モデルを推奨）を明示した上で、**/plan にそのまま渡せるコピー用ブロック** を作成します。
  - **重要:** プロンプト全体を **4重バッククォート (````)** で囲んだコードブロックとして出力してください。
  - **出力イメージ:**
    > 議論がまとまりました。以下の内容をコピーして、モデルを **Gemini 3.1 Pro (Thinking)** に切り替えてから `/plan` を実行してください。
    > ````markdown
    > /plan 
    > 
    > 【Brainstorming Note の合意事項】
    > [ここに note の内容を元にした設計指針や要件を要約]
    > ````

---

**システム命令:**
このフェーズでは、すぐに「正解」を出そうとしないでください。ユーザーとの対話を通じて、より良い設計を共創することに集中してください。コードスニペットの提示は構いませんが、あくまで「例」として提示し、実際のファイルへの適用は `/plan` 後の実装フェーズで行います。
