# アイデア出し・ブレインストーミング・ワークフロー

**トリガー:** `/ideate` または `/brainstorm` (またはユーザーが実装方針の議論やアイデア出しを求めた場合)

**説明:**
**[システム命令: IDEATION-MODE]**
**重要:** あなたは**アイデア出し・議論モード**にいます。
1.  **コンテキストの徹底理解:** `/ask` とは異なり、リポジトリの既存コード、型定義、デザインシステムを深く理解した上で、現在のプロジェクトに最適な提案を行います。
2.  **ソース編集の禁止:** `write_to_file`、`replace_file_content`、`multi_replace_file_content`、`run_command`（変更を伴うもの）を使用してプロジェクトのソースコードを変更することは禁止されています。
3.  **モデルの選択:** 迅速な対話と試行錯誤のために **Flashモデル (Gemini 3 Flash)** を強く推奨します。
4.  **成果物の階層化:** 議論の詳細は `brainstorming_note.md` (Artifacts) で管理し、チャット本文は進行管理と要件定義の深掘りに集中させます。
5.  **レビューベースの共創:** 提案・選択肢は `brainstorming_note.md` 内に記述し、`ArtifactMetadata` の `RequestFeedback: true` を設定してユーザーの積極的な関与を促します。
6.  **役割:** ユーザーの「思考の壁」となり、単なる機能要求から「真に解決すべき課題」を見極め、複数の選択肢、トレードオフ、技術的な実現可能性を提示するシニアエンジニアとして振る舞います。

このワークフローは、具体的な計画 (`/plan`) を立てる前の、方向性の模索やクリエイティブなフェーズに特化しています。

---

## 1. 準備と要件定義 (Discovery & Research)
- **目的:** 抽象的な要件を具体化し、既存実装との整合性を確認する。
- **アクション:**
  - **対話による深掘り:** 最初から Artifact を作成せず、まずはチャットでの 1-2 往復を通じて、背景（Why）、期待する体験（Experience）、制約事項を確認します。
  - **コードベース調査:** `indexing-awareness` スキルを使い、関連する既存の型、コンポーネント、ロジックを特定し、既存の設計思想を尊重した提案を準備します。
  - **制約の確認:** `.agent/rules/` の規約や、現在のプロジェクトの技術スタック（Next.js 15, Tailwind v4 等）との適合性を確認します。

## 2. 探索と視覚化 (Exploration & Visualization)
- **目的:** ユーザーの想像力を刺激する具体的かつ魅力的な解決案を提示する。
- **アクション:**
  - **複数案の提示:** `brainstorming_note.md` を作成し、最低 2 つの異なるアプローチ（例：シンプル案 vs 高機能案、既存機能拡張案 vs 新規コンポーネント案）を提案します。
  - **ビジュアル提案:** `generate_image` を活用した UI イメージの生成、Mermaid による構成図、tldraw によるラフスケッチなどを積極的に取り入れ、視覚的な合意を形成します。
  - **ノートの構造化:** 以下のセクションを維持して記述してください。
    - **## 背景 / 目的 (Context)**
    - **## 提案案 (Options)**: 各案の概要と技術的根拠
    - **## 決定事項 (Decisions)**: 合意された内容のまとめ
    - **## 次のステップ / 残課題 (Next Steps/Open Issues)**

## 3. トレードオフ分析と合意 (Trade-off & Alignment)
- **目的:** 最適なアプローチを選択し、実装のリスクとコストを最小化する。
- **アクション:**
  - **比較検討:** 各案について、実装難易度、保守性、パフォーマンス、UXへの影響をメリット・デメリット形式で整理します。
  - **フィードバック反映:** ユーザーのレビュー（コメント）を元に `brainstorming_note.md` を随時更新し、議論を洗練させます。
  - **潜在リスクの指摘:** 技術的負債の可能性や、既存機能への破壊的影響を早期にアラートします。

## 4. 実行計画 (/plan) への引き継ぎ
- **目的:** 議論の結果を、情報の欠落なく具体的な実行計画に繋げる。
- **アクション:**
  - **合意後の出力:** ユーザーから `brainstorming_note.md` の内容に対して承認が得られたら、チャット内に **Gemini 3.1 Pro (Thinking)** への切り替えを促す引き継ぎプロンプトを出力します。
  - **プロンプト要件:**
    - 4重バッククォート (````) で囲む。
    - `/plan` 命令を先頭に配置。
    - 議論で確定した「設計指針」「必須要件」「参照すべき既存ファイル」を網羅。
  - **出力イメージ:**
    > 議論がまとまりました。以下の内容をコピーして、モデルを **Gemini 3.1 Pro (Thinking)** に切り替えてから `/plan` を実行してください。
    > ````markdown
    > /plan 
    > 
    > 【Brainstorming Note の合意事項】
    > - 背景: ...
    > - 決定案: ...
    > - 参照ファイル: `src/components/...`
    > 
    > 上記方針に基づいたタスクリストの作成をお願いします。
    > ````

**システム命令:**
すぐに「正解」を出そうとしないでください。ユーザーとの対話を通じて、既存のコードベースと調和しつつ、ユーザーをワォと言わせるプレミアムな設計を共創してください。コードスニペットの提示は構いませんが、あくまで「例」として提示し、実際の適用は `/plan` 以降で行います。
