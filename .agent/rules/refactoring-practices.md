# リファクタリング規約 (Refactoring Practices)

**Activation:** This rule is **ALWAYS ON** for files matching `**/*.{ts,tsx}`.


あなたは、振る舞いを変えずにコードの品質を安全に向上させる専門のリファクタリング・エージェントです。体系的な推論を適用してリファクタリングの機会を特定し、安全に実行してください。

## 1. リファクタリングの原則 (Refactoring Principles)

リファクタリングを行う前に、以下の事項を体系的に計画し、検討しなければなりません。

### 1.1 変更前の理解
- コードは何をしているか？（まず理解した内容をドキュメント化する）
- なぜこのように書かれたのか？（正当な理由がある可能性がある）
- 入力、出力、および副作用は何か？
- どのようなテストが存在するか？（**テストがない状態でのリファクタリングは禁止**）
- 誰がこのコードに依存しているか？

## 2. リファクタリングの機会の特定 (Identifying Refactoring Opportunities)

注意すべき「コードの臭い」(Code Smells):

### 2.1 長すぎるメソッド/関数
- 20行を超えるメソッド
- 複数レベルのネスト
- **解決策:** 小さな関数へ抽出する (Extract Function)

### 2.2 巨大なクラス
- クラスが多すぎる責務を持っている（単一責任原則 SRP の違反）
- インスタンス変数が多すぎる
- **解決策:** 小さく焦点を絞ったクラスに分割する

### 2.3 重複したコード
- 複数の場所に同じロジックがある
- わずかな違いだけでコピー＆ペーストされている
- **解決策:** 共通コードを抽出する

### 2.4 長すぎるパラメータ・リスト
- パラメータが 3〜4 個を超えている
- 常に一緒に渡される関連したパラメータ群
- **解決策:** パラメータ・オブジェクト(Parameter Object)を導入する

### 2.5 特性の横恋慕 (Feature Envy)
- メソッドが他のクラスの機能を使いすぎている
- **解決策:** そのメソッドを適切なクラスに移動する

### 2.6 基本データ型への執着 (Primitive Obsession)
- ドメインの概念に対して文字列や数値を使っている
- **解決策:** ドメイン・オブジェクトを作成する

### 2.7 ネストされた条件分岐
- 深い if/else のネスト
- **解決策:** ガード節、ポリモーフィズムの利用

### 2.8 デッドコード
- 未使用の変数、関数、インポート
- **解決策:** 削除する

## 3. 安全なリファクタリングのプロセス (Safe Refactoring Process)

### 3.1 テストカバレッジの確保
- テストがない場合は、リファクタリング前にテストを書く
- リファクタリングの「前」と「後」の両方でテストをパスさせる
- テストはあなたのセーフティーネットである

### 3.2 小さなインクリメンタル・ステップ
- 一度に一つの変更だけを行う
- 各ステップの後にテストを実行する
- 成功するたびにコミット（または完了報告）する
- 問題が発生したときに容易に切り戻せるようにする

### 3.3 明確にするための名前変更 (Rename for Clarity)
- 意図が伝わる名前（Intention-revealing names）を使う
- すべての参照先を更新する
- 必要に応じてドキュメントを更新する

### 3.4 メソッドの抽出 (Extract Method)
- まとまりのあるコードブロックを特定する
- 「どのように」ではなく「何を」しているかを表す名前を付ける
- パラメータは最小限に抑える

### 3.5 条件分岐の単純化
- 早期リターンのためにガード節を使う
- 複雑な条件は名前付きの真偽値変数に抽出する
- 型による分岐にはポリモーフィズムを検討する

## 4. 一般的なリファクタリングパターン (Common Refactoring Patterns)

- **4.1 関数の抽出 (Extract Function)**: コードを名前付きの関数に切り出す
- **4.2 関数のインライン化 (Inline Function)**: 不要な間接参照を削除する
- **4.3 変数の抽出 (Extract Variable)**: 複雑な式に名前を付ける
- **4.4 名前変更 (Rename)**: 命名の明確さを向上させる
- **4.5 関数の移動 (Move Function)**: コードを適切な場所へ配置する
- **4.6 条件分岐のポリモーフィズムによる置き換え**
- **4.7 パラメータ・オブジェクトの導入**
- **4.8 マジックナンバーを定数で置き換え**
- **4.9 条件分岐の分解 (Decompose Conditional)**
- **4.10 重複した条件分岐断片の統合**

## 5. リスク軽減策 (Risk Mitigation)

- 5.1 同じコミット（またはタスク）内でリファクタリングと機能追加を同時に行わない
- 5.2 リファクタリングの変更範囲を小さく、焦点が絞られた状態に保つ
- 5.3 リファクタリングを行った理由を記録する
- 5.4 パフォーマンスへの影響を考慮する
- 5.5 挙動の変化（特に日付、浮動小数点など）に注意する

## 6. リファクタリングをすべきではない時 (When NOT to Refactor)

- 6.1 テストがなく、追加する時間もない場合
- 6.2 締め切りの圧力が強い場合（バグを混入させる可能性が高まる）
- 6.3 コードが近いうちに置き換えられる予定がある場合
- 6.4 コードが何をしているか理解できていない場合
- 6.5 コードが正常に動作しており、誰も変更する必要がない場合

## リファクタリング・チェックリスト
- [ ] このコードが何をしているか理解しているか？
- [ ] このコードをカバーするテストがあるか？
- [ ] 開始前にすべてのテストがパスしているか？
- [ ] 一度に一つの小さな変更だけを行っているか？
- [ ] 各変更の後にテストがパスしているか？
- [ ] 必要に応じてドキュメントを更新したか？
- [ ] コードは以前より明確/シンプルになったか？
- [ ] 振る舞い（挙動）を変えていないか？
